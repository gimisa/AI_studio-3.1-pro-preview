# Focus: Decoupling, input validation, and code modularity.
# ==============================================================================
PILLAR_2_ARCHITECTURE_AND_CODE:

  - Key: Master_Engine_Separation
    Reference: "Clean Architecture (Robert C. Martin)"
    Rule: "Code must follow a Master/Engine paradigm. 'Master' handles I/O, configuration parsing, and system calls. 'Engine' is a pure, agnostic library containing only computational logic."
    Rationale: "Prevents monolithic spaghetti code and allows isolated testing."
    Importance: CRITICAL

  - Key: Pure_Engine_Requirement
    Reference: "Separation of Concerns (Edsger W. Dijkstra)"
    Rule: "The computational Engine MUST NEVER contain hardcoded data, mock data, or embedded business rules. Rules belong in external YAML/JSON configurations."
    Rationale: "Hardcoding mocks is a severe anti-pattern that leads to technical debt."
    Importance: CRITICAL

  - Key: Strict_IO_Contract_Validation
    Reference: "Design by Contract (Bertrand Meyer) & Fail-Fast Principle"
    Rule: "Data anomalies (type mismatch, missing fields) MUST trigger an explicit `raise Exception` at the module boundary (Input). NEVER mask failures with default values (e.g., returning 0 or None) or silent `continue` blocks."
    Rationale: "Robustness comes from breaking execution at the exact point of data corruption, forcing upstream correction."
    Importance: CRITICAL

  - Key: Native_IO_Over_Heavy_Libraries
    Reference: "Occam's Razor (Law of Parsimony)"
    Rule: "For reading/writing raw or untrusted data, use native Python mechanisms (`with open`) and robust parsing logic. The use of Pandas for raw I/O pipelines is STRICTLY FORBIDDEN."
    Rationale: "Heavy dataframes mask underlying parsing errors and bloat dependencies unnecessarily."
    Importance: HIGH

  - Key: Exhaustive_Try_Blocks
    Reference: "Defensive Programming & Python PEP 8"
    Rule: "A `try` block MUST ALWAYS be paired with an `except` and/or `finally` block to handle garbage collection or specific error routing."
    Importance: CRITICAL
    
  - key: "Python_Indentation_And_Style_Contract"
    Reference: "PEP 8 – Style Guide for Python Code (Guido van Rossum, Barry Warsaw, Nick Coghlan)"
    Rule: "All Python code output MUST use exclusively 4 spaces per indentation level. Code blocks intended for insertion into an existing file MUST be delivered with indentation that exactly matches the surrounding scope."
    Rationale: "Indentation is part of Python’s syntax."
    Importance: CRITICAL
